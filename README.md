## ğŸ¤” Why Not Just Use NoSQL?

### The Evaluation Process

Uber evaluated existing NoSQL solutions (Cassandra, MongoDB, etc.) but decided against them:

**Reasons for Building Custom Solution:**

| Decision Factor | Why Build Custom |
|----------------|------------------|
| **Operational Trust** | Uber didn't have experience running NoSQL at scale |
| **MySQL Expertise** | Team already understood MySQL operations deeply |
| **Consistency Needs** | NoSQL eventual consistency complicated app logic |
| **Control** | Needed specific features: immutability, triggers, versioning |
| **Maturity at Scale** | Existing solutions not battle-tested at Uber's scale |

> **Key Insight**: Sometimes building on top of proven technology (MySQL) is safer than adopting new technology (NoSQL databases) when you don't have operational expertise.

### How Triggers Work

**1. The Trigger API**

```python
from schemaless import trigger

@trigger(instance='mezzanine', column='BASE')
def bill_rider(row_key):
    """
    Called automatically when a cell is written to BASE column
    
    row_key = trip UUID
    """
    # 1. Check if already processed (idempotency)
    status = schemaless.get(row_key, 'STATUS')
    if status.get('is_completed'):
        return  # Already billed, skip
    
    # 2. Get trip data
    trip = schemaless.get(row_key, 'BASE')
    
    # 3. Charge credit card
    result = credit_card_provider.charge(
        trip['rider_id'], 
        trip['fare']
    )
    
    # 4. Record success
    if result.success:
        schemaless.put(
            row_key, 
            'STATUS', 
            {'is_completed': True, 'charge_id': result.id}
        )
```

### Architectural Decisions Summary

| Decision | Rationale | Trade-off |
|----------|-----------|-----------|
| **MySQL over NoSQL** | Team expertise, operational maturity, proven reliability | Less "cool factor", more custom code needed |
| **Immutable cells** | Simplifies concurrency, provides audit trail, easier debugging | Disk space (minor with storage costs) |
| **Append-only writes** | No overwrite contention, linear disk I/O, faster | Can't truly delete (must mark as deleted) |
| **Eventually consistent indexes** | 4x faster writes, no distributed transactions | Stale reads possible for seconds |
| **Denormalized indexes** | Single-shard queries, much faster | Duplicate storage (acceptable trade-off) |
| **Fixed shard count** | No rehashing on growth, operational simplicity | Must choose wisely upfront |
| **Buffered writes** | High write availability during outages | Temporary read inconsistency |
| **Trigger framework** | Decoupled services, independent scaling | Requires idempotent handlers, eventual processing |

### What Worked Well

âœ… **Gradual Migration**
- Dual-writes provided safety net
- Shadow reads caught issues before impact
- Feature flags enabled instant rollback
- No downtime, no data loss

âœ… **Building on MySQL**
- Leveraged existing expertise
- Inherited battle-tested durability
- Used InnoDB caching and indexes
- Operational tooling already existed

âœ… **Immutable Data Model**
- Simplified concurrency (no locks needed)
- Complete audit trail for debugging
- Enables time-travel queries
- Easier to reason about

âœ… **Trigger Framework**
- Decoupled billing, analytics, notifications

## Summary

Uber's Schemaless demonstrates that sometimes the best solution is **building on proven technology** rather than adopting new paradigms. By layering a custom datastore on MySQL, they achieved:

- **Horizontal scalability** (4096 shards)
- **High availability** (master-minion replication)
- **Operational simplicity** (leveraging MySQL expertise)  
- **Developer velocity** (schemaless flexibility)
- **Event-driven architecture** (trigger framework)

The migration from Postgres to Schemaless - handling billions of trips with zero downtime - showcases world-class engineering: careful planning, gradual rollout, extensive validation, and the discipline to get it right.

Later optimizations (Python â†’ Go rewrite) show that even production systems can be dramatically improved without disruption when proper validation and rollout strategies are employed.

**Key Takeaway**: Great engineering isn't always about using the newest technology. It's about understanding your constraints, making informed trade-offs, and executing carefully.
